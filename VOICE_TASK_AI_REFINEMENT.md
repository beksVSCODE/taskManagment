# ИИ Уточнение Парсинга Голосовых Задач

## Описание

Реализовано двухэтапное парсирование голосовых команд с интеллектуальным уточнением через ИИ:

### Этап 1: Rule-Based Парсинг (всегда)
- Извлечение полей по ключевым маркерам (исполнитель, описание, название, дедлайн, приоритет)
- Регулярные выражения для быстрого распознавания структуры
- Санитизация для предотвращения перекрёстного загрязнения полей

### Этап 2: ИИ Уточнение (если включено)
- ИИ получает уже парсённые данные как контекст
- Интеллектуальное улучшение извлечённых значений
- Исправление опечаток в именах исполнителей
- Очистка названий задач от мусора
- Правильное определение приоритета и дедлайнов

## Архитектура

```
Запись речи → STT (браузер/сервер) → Транскрипт
                                         ↓
                              Rule-Based Парсинг
                                         ↓
                      Парсённые данные (draft_v1)
                                         ↓
                      LLM Уточнение (refineWithAI)
                                         ↓
                      Улучшенные данные (draft_v2)
                                         ↓
                     Пользователь подтверждает
                                         ↓
                        TaskService.createTask()
```

## Ключевые Компоненты

### VoiceTaskService.parse()
1. Парсит transcript по регулярным выражениям SECTION_*
2. Вызывает `voiceIntentParserService.refineWithAI()`
3. Если LLM вернул данные, использует их для улучшения draft
4. Возвращает полный draft с confidence и warnings

### ProviderVoiceIntentParserService.refineWithAI()
- Двухэтапный процесс с предварительно извлечёнными данными
- OpenAI: temperature=0.3 для контролируемого улучшения
- Gemini: JSON Schema для структурированного вывода
- Fallback: если ИИ недоступен, используются rule-based результаты

### Поддерживаемые Поля

| Поле | Маркеры | Пример |
|------|---------|--------|
| Исполнитель | `исполнитель`, `исполниетль` | "исполнитель Иван Петров" |
| Название | `название`, `название задачи`, `задача` | "название Подготовить отчёт" |
| Описание | `описание` | "описание Включить анализ продаж за квартал" |
| Дедлайн | `дедлайн`, `дедлайн до`, `срок`, `срок до` | "дедлайн до 28 апреля" |
| Приоритет | `приоритет`, `приоритетность` | "приоритет высокий" |

## Конфигурация

### application.properties
```properties
# Провайдер STT (speech-to-text): client, openai, gemini
voice.stt.provider=client

# Провайдер парсинга: rule, openai, gemini
voice.parse.provider=rule

# Для OpenAI
voice.parse.openai.base-url=https://api.openai.com/v1
voice.parse.openai.api-key=${OPENAI_API_KEY}
voice.parse.openai.model=gpt-4o-mini

# Для Gemini
voice.parse.gemini.base-url=https://generativelanguage.googleapis.com/v1beta
voice.parse.gemini.api-key=${GEMINI_API_KEY}
voice.parse.gemini.model=gemini-2.0-flash
```

### Переменные окружения
```bash
export OPENAI_API_KEY="sk-..."
export GEMINI_API_KEY="..."
export VOICE_PARSE_PROVIDER=openai
```

## Примеры Использования

### Пример 1: Простая голосовая команда
```
"Назначь Иван исполнителем, описание: подготовить отчёт, дедлайн до 28 апреля, приоритет высокий"
```

**Результат парсинга:**
- assigneeRaw: "Иван" → assigneeId: 5 (fuzzy matching)
- title: "подготовить отчёт" (автоматически из контекста)
- description: "подготовить отчёт"
- dueDate: "2026-04-28"
- priority: "HIGH"

### Пример 2: Свободная речь с опечатками
```
"Для Иванна, сделай подготовку отчета про продажи, нужно до пятницы, оченьважно"
```

**Rule-Based результат:**
- assigneeRaw: "Иванна" (опечатка)
- title: "подготовку отчета про продажи"

**После ИИ уточнения:**
- assigneeRaw: "Иван" (исправлена опечатка)
- title: "Подготовить отчёт по продажам"
- description: "Анализ продаж"

## Отладка

### Проверка работы парсинга
```bash
curl -X POST http://localhost:8080/api/projects/1/tasks/voice/parse \
  -F "audio=@recording.wav" \
  -F "transcript=назначь Иван исполнителем"
```

### Просмотр confidence и warnings
Ответ содержит:
- `confidence`: 0-1, вероятность корректности парсинга
- `warnings`: массив предупреждений
- `missingFields`: какие поля не удалось заполнить
- `assigneeCandidates`: список найденных исполнителей

## Производительность

- Rule-Based: <10ms
- LLM Refinement (OpenAI): 300-800ms
- LLM Refinement (Gemini): 200-600ms
- Общее время: ~1 сек для типичной команды

## Обработка Ошибок

1. Если LLM недоступен → используются rule-based результаты
2. Если парсинг не уверен → выводятся кандидаты для ручного выбора
3. Если confidence < 0.65 → добавляется предупреждение
4. Если ошибка критична → возвращается HTTP 400 с описанием
